For each enum, also export a Zod version. Example pattern:

```typescript
import { z } from 'zod';

export const EnvironmentValues = ['dev', 'test', 'prod'] as const;
export type Environment = (typeof EnvironmentValues)[number];
export const EnvironmentSchema = z.enum(EnvironmentValues);
```

Use this pattern for ALL enums. It gives us: the type for interfaces, the values array for iteration/validation, and the Zod schema for runtime checks.

### Zod Schemas

For each type, create a matching Zod schema in `schemas/`. The schema must:
- Use the enum Zod schemas defined in `enums/index.ts`
- Use `z.string().uuid()` for all `id` fields
- Use `z.string().datetime()` for all date fields (ISO 8601)
- Use `z.string().min(1)` for required string fields
- Use `z.string().min(10).max(1000)` for the `rationale` field on PolicyRule
- Use `z.string().email()` for email fields
- Use `z.record(z.unknown()).nullable()` for JSON fields
- Use `z.array(...)` for array fields
- Export both the schema and the inferred type: `export type AgentInput = z.infer<typeof AgentSchema>`

Also create "input" schemas (for API request validation) that omit server-generated fields:
- `AgentCreateSchema` — omits id, tenant_id, created_at, updated_at, lifecycle_state (defaults to 'active')
- `AgentUpdateSchema` — all fields optional except id
- `PolicyRuleCreateSchema` — omits id, tenant_id, policy_version, created_at, updated_at, is_active
- `PolicyRuleUpdateSchema` — all fields optional except id
- `ApprovalDecisionSchema` — `{ approver_name: string, decision_note?: string }`
- `EvaluateRequestSchema` — matches EvaluateRequest interface
